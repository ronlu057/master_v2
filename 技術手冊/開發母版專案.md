## 1. 母版概述

### 1.1 設計目標

一套母版同時具備「快速套版」與「深度客製」兩端的能力。實務上專案需求落在一條光譜上 —— 從幾乎不改的套版站，到整站手刻。本母版用**一個設定值**決定當前專案落在光譜何處，其餘程式碼共用。

### 1.2 共用基底

無論哪種專案類型，以下三層完全共用、不需重寫：

| 層級 | 內容 |
|---|---|
| **API 串接層** | `composables/useApi.js` — 統一處理[一般網站白皮書](./一般網站白皮書.md) §2.1 回應格式 |
| **元件庫** | 頁首/頁尾/選單/Banner/彈跳廣告/Block 模組/新聞卡片… |
| **頁面路由** | 首頁、新聞、關於、聯絡、搜尋、FAQ、服務據點、購物 等 |

差異在於「**首頁與內頁各用多少手刻、多少 block 模組**」，由專案類型決定。

## 4. 五種專案類型

由 `.env` 的 `NUXT_PUBLIC_PROJECT_TYPE` 決定，五選一：

| 類型 | 設定值 | 首頁 | 內頁 | 購物 | 適用情境 |
|---|---|---|---|---|---|
| **模組件套版** | `module` | block 模組組裝 | block 模組組裝 | ✗ | 標準形象站，後台 cipage 編輯內容 |
| **客製首頁** | `custom-home` | 手刻 + block 模組組裝 | block 模組組裝 | ✗ | 首頁要特殊視覺、內頁走標準模組 |
| **全部客製開發** | `full-custom` | 手刻 + block 模組組裝 | 手刻 | ✗ | 整站皆需獨特設計 |
| **購物類型** | `shop` | 手刻 + block 模組組裝 | block 模組組裝 | ✓ | 電商站，啟用購物車與商品頁 |
| **臨時站** | `temp` | 固定模板 | 隱藏 | ✗ | 「即將開幕」、活動過渡頁、暫代站 |

### 4.1 功能矩陣

每種類型對應三個旗標（定義於 `app.config.js`）：

| 旗標 | 意義 |
|---|---|
| `customHome` | 首頁採手刻版型、並可嵌入 block 模組（`true`）；或整頁 block 模組組裝（`false`） |
| `customPages` | 內頁手刻（`true`）或 block 模組組裝（`false`） |
| `shop` | 是否啟用購物模組 |
| `minimal` | 精簡模式（臨時站）：固定模板、單頁呈現、隱藏完整選單 |

### 4.2 切換方式

```bash
# 編輯 .env
NUXT_PUBLIC_PROJECT_TYPE=shop

# 重啟 dev server
npm run dev
```

切換後 `<html>` 會帶上 `data-project="shop"`，CSS 變數主題同步換色（見 §6、§9.4）。

### 4.3 加掛購物功能（強制購物開關）

客戶為非 `shop` 類型、但又需要單獨啟用購物車 / 我的最愛等 UI（**混合需求**）時，可用獨立開關 `NUXT_PUBLIC_ENABLE_SHOP` 強制啟用：

```bash
# 編輯 .env
NUXT_PUBLIC_PROJECT_TYPE=module     # 維持原本專案類型（不切換整站架構）
NUXT_PUBLIC_ENABLE_SHOP=true        # 強制開啟購物 UI

# 重啟 dev server
npm run dev
```

對應 `useProject()` 兩個旗標差別：

| 旗標 | 啟用條件 | 用途 |
|---|---|---|
| `isShop` | 僅 `NUXT_PUBLIC_PROJECT_TYPE=shop` 時為 `true` | **架構決策**：首頁版型派發、shop layout、商品路由等 |
| `enableShop` | `isShop` 為 `true` **或** `NUXT_PUBLIC_ENABLE_SHOP=true` | **UI 顯示**：購物車、我的最愛、收藏等元件是否渲染 |

> [!IMPORTANT]
> **元件內請用 `enableShop` 控制購物 UI 顯示**（如 `<NuxtLink v-if="enableShop" to="/shop/cart">`），不要直接用 `isShop`。
> `isShop` 是「整站是不是購物站」，`enableShop` 是「要不要顯示購物功能」—— 後者比前者寬鬆，混合需求才能 work。

範例（[`components/headers/Header01.vue`](../components/headers/Header01.vue)）：

```vue
<script setup>
const { enableShop } = useProject()
</script>

<template>
  <template v-if="enableShop">
    <NuxtLink to="/shop/cart">購物車</NuxtLink>
    <NuxtLink to="/shop/favorite">我的最愛</NuxtLink>
  </template>
</template>
```

## 6. 專案類型切換機制

三個檔案協同運作：

```
.env (NUXT_PUBLIC_PROJECT_TYPE)
        ↓ 注入 runtimeConfig.public.projectType
nuxt.config.js
        ↓ 讀取
composables/useProject.js  ←─ 對照 app.config.js 的 projectTypes 矩陣
        ↓ 回傳 { type, customHome, customPages, isShop, isMinimal }
頁面 / 元件
```

### 6.1 在頁面/元件中使用

```javascript
const { type, customHome, isShop, isMinimal } = useProject()
```

### 6.2 首頁分流（`pages/index.vue`）

```vue
<template>
  <HomeMinimal v-if="isMinimal" />       <!-- 臨時站 -->
  <HomeCustom  v-else-if="customHome" /> <!-- 客製首頁 / 全客製 / 購物 -->
  <HomeModule  v-else />                 <!-- 模組件套版 -->
</template>
```

新增專案類型：在 `app.config.js` 的 `projectTypes` 加一筆、`.env` 改設定值即可。

## 7. API 串接層與 Mock JSON

### 7.1 統一串接層

所有 API 請求一律經 `composables/useApi.js`，它負責解開[一般網站白皮書](./一般網站白皮書.md) §2.1 的回應格式 `{ status, data: { payload, meta } }`，頁面拿到的直接是 `payload`。

```javascript
// SSR 資料獲取（頁面 / 元件 setup 階段）
const { data } = await useApiData('/home/view')

// 命令式請求（表單送出等 POST 互動）
const res = await $api('/contact/submit', { method: 'POST', body })
```

### 7.2 Mock JSON ↔ 真實後端切換

由 `.env` 的 `NUXT_PUBLIC_API_BASE` 控制：

| `NUXT_PUBLIC_API_BASE` | 行為 |
|---|---|
| 留空 | 打本地 mock JSON（`server/routes/api/[...slug].js`） |
| 填後端網域 | 打真實後端，回應格式不變，**前端程式碼無需修改** |

前端開發階段一律留空，用 mock 資料開發版面與互動；後端就緒後再填網域。

### 7.3 Mock 層結構

- `server/mock/db.js` — 模擬資料（對應白皮書各 controller 的 payload）
- `server/routes/api/[...slug].js` — 派發器，依路徑/方法回傳對應 mock，並包成白皮書回應格式

新增 mock endpoint：在 `db.js` 補資料、在派發器 `switch` 加 `case` 即可。

## 8. Block 模組系統

Block 系統讓後端一次回完整資料、前端只負責 render（[一般網站白皮書](./一般網站白皮書.md) §3）。模組件套版整頁皆由 block 組裝；客製首頁／全客製／購物則在手刻版型中嵌入 block 模組（見 `components/HomeCustom.vue`）。

### 8.1 資料結構

```
payload.slots[] → slot.blocks[] → block.item.rows[]
```

每個 `block` 帶 `type`（決定套哪個元件）、`title`、`item.rows`（內容固定取此）。

> ⚠️ **例外：`banner` block**。模組首頁的 banner 內容**不**內嵌在 `/home/view` 聚合裡，改由獨立 **Banner API `/banner/home`** 提供 —— `BlockBanner` 派發器自己 `useApiData('/banner/home')` 取資料（rows + 背景影片 + 跑馬燈 news）。聚合裡的 banner block 只留 `{ "type": "banner" }` 當定位。模擬「未來後端會有多支獨立 API（Header / Banner …）」，不是單一大聚合。詳見 §12.3。

### 8.2 BlockRenderer

`components/BlockRenderer.js`（render function）維護 `block.type → 元件` 對照表，未知 type 直接 skip：

| `block.type` | 對應元件 |
|---|---|
| `banner` | `BlockBanner`（輪播） |
| `intro` | `BlockIntro`（圖文左右排版） |
| `news` | `BlockNews`（新聞卡片列） |
| `push` / `other_ad` | `BlockGeneric`（通用卡片格） |

新增 block 樣式：建立 `BlockXxx.vue`、在 `BlockRenderer.js` 的對照表加一筆。

> ⚠️ Nuxt 自動引入的元件**無法**用 `resolveComponent('字串')` 取得；
> Block 元件須在 `BlockRenderer.js` 以 `import` 明確引入後放入對照表。
>
> 💡 `BlockRenderer` 刻意用 **render function（`.js` functional component）** 而非 template 的 `v-for`/`v-if`：
> 後者會被編譯成 Vue Fragment，SSR 時每個 Fragment 都吐出 `<!--[-->` / `<!--]-->` 邊界註解。
> render function 直接把 block 掛成 wrapper element 的 children 陣列、無 Fragment，
> **維持 SSR/SEO，但 dispatch 層不再產生 fragment 註解**（這是 §9.1 「元件一律 `<script setup>`」的刻意例外）。

## 9. 開發與編程規範

### 9.1 Vue 元件規範

1. 統一 `<script setup>` 語法。
2. **不使用 TypeScript**。
3. 元件命名 **PascalCase** 且多字詞（如 `AppHeader.vue`、`NewsCard.vue`）。
4. 善用 Nuxt 自動引入：`components/`、`composables/`、`stores/` 與 Vue API（`ref`、`computed`…）皆免手動 `import`。

### 9.2 狀態管理（Pinia）

- Store 集中於 `stores/`，採 **Setup Store** 風格。
- 範例：`useUiStore`（行動選單開關）、`useCartStore`（購物車）。

```javascript
// stores/useCartStore.js
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'

export const useCartStore = defineStore('cart', () => {
  const items = ref([])
  const count = computed(() => items.value.reduce((s, i) => s + i.qty, 0))
  function add(product, qty = 1) { /* ... */ }
  return { items, count, add }
})
```

### 9.3 資料獲取

- SSR 資料用 `useApiData`（包裝 `useFetch`）；POST 互動用 `$api`（包裝 `$fetch`）。
- 勿在 setup 外部調用非同步生命週期鉤子。

### 9.4 樣式與主題

> [!IMPORTANT]
> **全站樣式規範**
> - 一律使用 Sass：所有 `<style>` 必須加 `lang="scss"`；嚴禁純 CSS、嚴禁無 `lang` 的 `<style>`，專案內不得出現獨立 `.css` 檔。
> - 嚴禁標籤行內 `style="..."` **與 `:style="..."` 綁定**：行內樣式優先級最高、**無法被 media query 覆寫 → 破壞 RWD**。樣式寫進 `<style lang="scss" scoped>` 以 class 套用、動態切換用 `:class`。唯一例外＝執行期才算得出的動態幾何值（Swiper 進度條高度、flex `order`…）。詳見 [製作規範 §3.1.2](./製作規範.md#312-禁用行內樣式-style特別注意)。
> - 嚴禁 `!important`（全站不使用，無例外）。
> - 桌機（1024–2560）的字級／版面尺寸用流體 `fluid()` 等比縮放（[§9.5.1](#951-流體尺寸-fluid桌機-10242560-等比縮放)）。
> - **禁用 BEM `&__` 巢狀寫法** —— class 必須寫完整名稱（詳見 §9.4.1）。

- 主題色採 **CSS 變數**，定義於 `assets/styles/main.scss` 的 `:root`。
- 五種專案類型各有一組主題色，以 `[data-project='xxx']` 覆寫（`<html>` 屬性由 `app.vue` 設定）。
- 元件樣式用 `<style lang="scss" scoped>`，色彩一律引用 `var(--color-primary)` 等變數。

```scss
.btn { background: var(--color-primary); }
```

#### 9.4.1 禁用 BEM `&__` 巢狀寫法

❌ 不要用 `&` 父選擇器組 BEM child class：

```scss
// 禁
.card {
  padding: 20px;
  &__title { font-size: 20px; }    // ❌ 巢狀組合 child
  &__body  { color: #333; }        // ❌
}
```

✅ 必須寫完整 class 名稱：

```scss
// 推薦
.card       { padding: 20px; }
.card__title { font-size: 20px; }   // ✓ 完整名稱
.card__body  { color: #333; }       // ✓
```

**為什麼禁用**：

| 理由 | 說明 |
|---|---|
| **可搜尋性** | `grep .card__title` 直接找得到定義；用 `&__` 寫法要先找父 selector |
| **重構安全** | 改 `.card` 為 `.product-card` 時，`&__` 衍生 class 不會自動連動，容易斷裂 |
| **學習負擔低** | 新進工程師看完整 class 就懂，不用懂 `&` 上下文 |
| **編譯結果相同** | 兩種寫法輸出 CSS 完全一樣，無效能差異 |

> [!IMPORTANT]
> **判斷標準**：`&` 後面是「**衍生新 class 名稱**」就禁；「**附加既有條件**」就 OK。詳見 [製作規範 §3.1.1](./製作規範.md#311-禁用-bem-__-巢狀寫法)。
>
> | 寫法 | 判定 |
> |---|---|
> | `&:hover` / `&::before` / `&[data-page]` | ✅ OK（偽類 / 偽元素 / 屬性選擇器） |
> | `&.is-active` / `&.show` / `&.scroll` | ✅ OK（附加既有狀態 class） |
> | `&__xxx` / `&--xxx` / `&-large` | ❌ 禁 — 衍生新 class 名稱，難 grep |

### 9.5 RWD 斷點 mixin

10 個斷點 mixin 定義於 [`assets/styles/_variables.scss`](../assets/styles/_variables.scss)，由 [`nuxt.config.js`](../nuxt.config.js) 的 `additionalData` 自動注入到每個 `<style lang="scss">`，**元件內直接 `@include` 即可**，不需 `@use`：

斷點（max-width）：`2560 / 1920 / 1680 / 1440 / 1280 / 1200 / 1024 / 768 / 480 / 360`（共 10 個）；其中 **1200** 為 header 變更樣式主斷點，**2560** 對應 4K 大螢幕。

```scss
.box {
  width: 25%;
  @include rwd-1440 { width: 33%; }
  @include rwd-1200 { width: 50%; }
  @include rwd-768  { width: 100%; }
}
```

> [!IMPORTANT]
> 兩種寫法都可以，差別在 **數值是否落在標準斷點**：
> - **標準斷點** → 用 `@include rwd-XXX`（與全站連動，未來改值動一處即可）
> - **非標準斷點 / 非寬度條件**（如 `540px`、`orientation: landscape`、`print`）→ 直接 `@media`
>
> 唯一禁止：用 `@media (max-width: 1200px)` 這種**剛好等於標準斷點數字**的寫法 —— 改成對應的 `@include rwd-1200` 才能跟全站同步。

#### 9.5.1 流體尺寸 `fluid()`（桌機 1024–2560 等比縮放）

斷點 mixin 之外，桌機區間的「等比縮放」用 [`_variables.scss`](../assets/styles/_variables.scss) 的 **`fluid()`** 函式：以 **1920 設計稿**為基準，字級／版面寬高／間距／定位等隨視窗等比，全程 CSS `calc`（不寫死算完的數值）。

```scss
@function fluid($target, $min: 1024, $max: 2560) {
  @return clamp(
    calc(#{$target} / 1920 * 1024 * 1px),   // 下限＝1024 時的值
    calc(#{$target} / 19.2 * 1vw),          // 中段＝隨視窗等比；1920 時 = $target
    calc(#{$target} / 1920 * 2560 * 1px)    // 上限＝2560 時的值
  );
}
// 用法
font-size: fluid(56);
--container: fluid(1600);
```

- **1920 時 = 設計值**（與舊設定一致）；**1024–1920**、**1920–2560** 都等比；≤1024 維持 1024 值（交給手機版型）、≥2560 鎖住。
- 目標：**2560 與 1920 同比例、位置零落差**。哪些 px 要 `fluid()`、哪些維持固定（1–2px 細邊、小 UI 字 <20px、`box-shadow`、`@media` 區塊、`/admin` 後台…）詳見 [製作規範 §3.6](./製作規範.md#36-字級尺寸流體-rwd-規範)。

### 9.6 路由中介層（middleware）

`middleware/` 放路由中介層 —— 進入頁面前執行的攔截 / 守衛邏輯（登入驗證、轉址、權限檢查等）。

- **具名中介層**：`middleware/auth.js`，於頁面以 `definePageMeta({ middleware: 'auth' })` 套用。
- **全域中介層**：檔名加 `.global` 後綴（`middleware/xxx.global.js`），每次路由切換皆自動執行。

```javascript
// middleware/auth.js — 具名中介層範例
export default defineNuxtRouteMiddleware((to, from) => {
  // 不符合條件時導向其他頁
  // if (!isLoggedIn()) return navigateTo('/login')
})
```

### 9.7 工具函式（utils）

`utils/` 放**純 JavaScript 工具函式**（格式化、驗證、計算等不涉及 Vue 響應式的邏輯）。Nuxt 會**自動引入**，全站免 `import` 直接呼叫。

- 與 `composables/` 的分工：`composables/` 放含響應式 / 生命週期的組合式函式（`useXxx`）；`utils/` 放無狀態的純函式。

```javascript
// utils/formatPrice.js — 工具函式範例
export function formatPrice(n) {
  return 'NT$ ' + Number(n).toLocaleString()
}
// 任何元件 / 頁面可直接呼叫 formatPrice(1280)，不需 import
```

### 9.8 切版分工（AI 使用範圍）

> [!NOTE]
> **不建議用 AI 做像素級切版**
> 經實測，AI 從設計圖將視覺稿轉為精準 CSS 時，尺寸／樣式會跑掉 —— AI 只能由平面圖「目測估算」px／顏色／間距，無法逐像素還原。

視覺精準還原由人工處理；AI 專注於結構與邏輯：

| 由 AI 處理 | 由人工處理 |
|---|---|
| 母版架構、專案類型、header 派發等系統 | 像素級視覺還原 |
| RWD 斷點、元件結構組裝 | 精準尺寸／配色對稿 |
| Pinia、API／mock 串接、Block 系統 | 設計稿精修 |
| 白皮書文件、設定檔、git | — |

HTML 結構骨架可由 AI 建立，但 `<style>` 內的精準尺寸由人工調整。

### 9.9 Google Fonts 字型

字型於 [`nuxt.config.js`](../nuxt.config.js) 的 `app.head.link` 集中設定，會自動注入每個頁面的 `<head>`。**不採用 Google 早期的 `webfont.js` 載入器**（SSR 不友好、會多下載 28KB JS、首屏會閃白）。

```js
// nuxt.config.js
app: {
  head: {
    link: [
      { rel: 'preconnect', href: 'https://fonts.googleapis.com' },
      { rel: 'preconnect', href: 'https://fonts.gstatic.com', crossorigin: '' },
      {
        rel: 'stylesheet',
        href: 'https://fonts.googleapis.com/css2?family=…&display=swap',
      },
    ],
  },
},
```

**目前已載入**

| 字型 | 字重 |
|---|---|
| Noto Sans TC | 100, 200, 300, 400, 500, 600, 700, 800, 900 |
| Noto Serif TC | 200, 300, 400, 500, 600, 700, 800, 900 |
| Poppins | 100, 200, 300, 400, 500, 600, 700, 800, 900 |
| Roboto | 100, 300, 400, 500, 700, 900 |
| Gruppo | 400 |

**對應 SCSS 變數**（[`assets/styles/_variables.scss`](../assets/styles/_variables.scss)）—— 元件內請直接引用變數，避免到處寫死字型名稱：

```scss
$web_font: 'Noto Sans TC', sans-serif;        // 全站主字體
$title_font_en: 'Poppins', sans-serif;        // 標題英文
$title_font_cht: 'Noto Sans TC', sans-serif;  // 標題中文
$btn_font: 'Noto Sans TC', sans-serif;        // 按鈕字體
```

#### 9.9.1 新增或修改字型

當設計師指定新字型、或要調整字重時，請依序：

1. **到 [Google Fonts](https://fonts.google.com)** 選字型 → 點 **Get font** → **Get embed code** → 複製 `<link>` 內 `href` 的 URL 參數段，例如：
   ```
   family=Inter:wght@300;400;500;700&display=swap
   ```

2. **修改 [`nuxt.config.js`](../nuxt.config.js)** 內 `app.head.link` 最後一筆 stylesheet 的 `href` —— 用 `&family=` 串接新字型：
   ```js
   href: 'https://fonts.googleapis.com/css2?'
     + 'family=Noto+Sans+TC:wght@100;200;300;400;500;600;700;800;900'
     + '&family=Inter:wght@300;400;500;700'   // ← 新增這行
     + '&display=swap',
   ```

3. **更新 [`assets/styles/_variables.scss`](../assets/styles/_variables.scss)** 的字型變數（如要設為全站主字體）：
   ```scss
   $web_font: 'Inter', 'Noto Sans TC', sans-serif;
   ```

4. **重啟 dev server** —— `nuxt.config.js` 的變更不在 HMR 範圍。

5. **驗證** —— DevTools → Network → 確認 `fonts.googleapis.com/css2?…` 回 200，且 `Computed → font-family` 確實是新字型。

> [!IMPORTANT]
> **字型只在這兩個檔案管理**：URL 在 `nuxt.config.js`、別名變數在 `_variables.scss`。**禁止**在元件內 import Google Fonts CSS、或重複寫死 `font-family: 'Noto Sans TC'` —— 一律走 SCSS 變數，未來要改字型只動一處。

#### 9.9.2 移除字型

不再用的字型要主動移除以縮減首屏載入：
1. 從 `nuxt.config.js` 的 `href` 移除對應 `&family=...` 段。
2. grep 確認沒有 `_variables.scss` 變數或元件還在引用該字型名。

#### 9.9.3 為何用 `display=swap`

`&display=swap` 告訴瀏覽器：字型還沒載入完成時，**先用 fallback 顯示文字**，等載完再切換 —— 避免使用者看到空白文字（FOIT）。代價是字型切換瞬間會有微小視覺跳動（FOUT），但對使用者體驗整體較好。

全站採用 [`@nuxtjs/i18n`](https://i18n.nuxtjs.org/) 處理多語：清單集中設定、Header 自動渲染切換、cookie 自動記住使用者選擇。

#### 9.10.1 設定位置一覽

| 改什麼 | 改哪 | 重啟 dev? |

| URL 策略（是否帶 `/en/` 前綴） | [nuxt.config.js](../nuxt.config.js) 的 `i18n.strategy` | ✅ |

#### 9.10.3 元件內用法

```vue
<script setup>
// 取翻譯字串、目前語系、切換 function

</script>

<template>
  <!-- 1. 取翻譯字串 -->
  <h1>{{ t('site.title') }}</h1>
  <button>{{ t('btn.more') }}</button>

  <!-- 2. 取目前語系 -->

```

```vue
<button>{{ $t('btn.search') }}</button>
```

#### 9.10.4 Header 切換 UI 規範

語系切換 icon 與選單在任何元件（Header / Footer / 浮動工具列…）顯示時，**必須**判斷語系數量，符合下列規則才渲染：

- 語系數量 ≤ 1 → **不顯示** icon（無切換意義，多餘 UI）
- 語系數量 ≥ 2 → 才顯示 icon + 切換選單

範例（[`components/headers/Header01.vue`](../components/headers/Header01.vue)）：

```vue
<script setup>
const { locale, locales, setLocale } = useI18n()
const languages = computed(() =>
  locales.value.map((l) => ({ code: l.code, label: l.name })),
)
</script>

<template>
  <div v-if="languages.length > 1" class="navtool_icon">
    <LanguageIcon aria-label="語系" />
    <div class="lang_box">
      <button
        v-for="lang in languages"
        :key="lang.code"
        :class="{ 'is-active': lang.code === locale }"
        @click="setLocale(lang.code)"
      >
        {{ lang.label }}
      </button>
    </div>
  </div>
</template>
```

> [!IMPORTANT]
> **語系清單只在 `nuxt.config.js` 一處維護**。Header / Footer / 任何元件都用 `useI18n().locales` 動態取得，**禁止**在元件內硬編碼 `languages` 陣列。

#### 9.10.5 URL 策略選擇

| 策略 | URL 結果（defaultLocale = tw） | SEO | 適用 |
|---|---|---|---|
| `no_prefix`（目前） | `/about` 全語系共用 | 弱 | 後台規模小、靠 cookie 切換 |
| `prefix_except_default` | tw: `/about` / en: `/en/about` | 佳 | **推薦** SEO 友好 |
| `prefix` | 全部 `/tw/about` `/en/about` | 佳 | 對所有語系 SEO 均等 |
| `prefix_and_default` | 同 `prefix`，default 也接受無前綴 | 中 | 過渡用 |

切換策略時，記得每頁加 `useLocaleHead()` 產生 `<link rel="alternate" hreflang>`：

```vue
<script setup>
useLocaleHead()  // 自動為當前頁的所有語系變體產生 SEO meta
</script>
```

#### 9.10.6 與 mock `/i18n` endpoint 的關係

專案另有 [`composables/useSite.js`](../composables/useSite.js) 的 `useDict()` 從 `/api/i18n` 拉動態字典 —— 這是**從後台拿翻譯**的機制，跟 `@nuxtjs/i18n` 的本地字典**並行存在**：

| 用途 | 用什麼 |
|---|---|
| 介面固定文字（按鈕、Menu、提示） | `useI18n().t('key')` ← 推薦 |
| 後台可改的動態文字（公告、橫幅文案） | `useDict().t('key')` |

> [!TIP]
> 大部分時候用 `useI18n().t()` 就好。`useDict()` 只在「客戶要能後台改文字」時才用。

### 9.11 平滑捲動（Lenis + GSAP）

全站採用 [Lenis](https://github.com/darkroomengineering/lenis) 接管 wheel / touch 捲動，加入慣性 + 緩動。同時整合 [GSAP](https://gsap.com/) / [ScrollTrigger](https://gsap.com/docs/v3/Plugins/ScrollTrigger/)，讓動畫與捲動位置同步。

**設定位置**：[`plugins/lenis.client.js`](../plugins/lenis.client.js)（`.client.js` 後綴 → 僅 client 端載入，SSR 階段不執行）

```js
const lenis = new Lenis({
  duration: 1.2,                                          // 捲動緩衝時間（秒），越大越「滑」
  easing: (t) => Math.min(1, 1.001 - Math.pow(2, -10 * t)), // expoOut 緩動曲線
  smoothWheel: true,                                       // 滑鼠滾輪平滑
  // syncTouch 預設 false → 行動裝置走系統原生慣性，避免雙重平滑感
})

// 用 GSAP ticker 驅動 Lenis（不用 requestAnimationFrame）
gsap.ticker.add((time) => lenis.raf(time * 1000))
gsap.ticker.lagSmoothing(0)
lenis.on('scroll', ScrollTrigger.update)
```

#### 9.11.1 自動行為（不需手動觸發）

- ✅ **全站接管捲動** —— 任何頁面、任何長頁面都會自動有平滑效果
- ✅ **路由切換歸頂** —— `router.afterEach` 內呼叫 `lenis.scrollTo(0, { immediate: true })`，新頁面不會「滑進」
- ✅ **`window.scroll` 事件仍會 fire** —— 元件內的 `window.addEventListener('scroll', ...)` 無需修改（如 [`Header01.vue`](../components/headers/Header01.vue) 的捲動偵測）
- ✅ **第三方 modal 自動鎖背景** —— `MutationObserver` 監聽 `<body>` class，當 LightGallery / Magnific Popup 等套件加上鎖捲動 class 時自動 `lenis.stop()`（詳見 §9.11.5）

#### 9.11.2 進階：程式式捲動

任何元件 / 頁面取得 lenis instance：

```vue
<script setup>
const { $lenis } = useNuxtApp()
</script>

<template>
  <!-- 捲到頂部（例：點 Logo 回頂、回到頂部按鈕） -->
  <button @click="$lenis.scrollTo(0)">↑ 回頂</button>

  <!-- 捲到指定 selector（取代錨點 #xxx，可控速度） -->
  <button @click="$lenis.scrollTo('#about', { duration: 1.5 })">關於我們</button>

  <!-- 捲到指定 DOM 元素 -->
  <button @click="$lenis.scrollTo(myRef, { offset: -100 })">滑到區塊</button>
</template>
```

`scrollTo` 第二參數常用選項：

| 選項 | 說明 |
|---|---|
| `duration` | 緩衝時間（秒）；預設沿用 Lenis 全域設定 |
| `offset` | 目標位置上下偏移（px）；負值預留 Header 高度，正值往下 |
| `immediate` | `true` = 不平滑、立即跳到位置；`false`（預設）= 用設定的緩動 |
| `lock` | `true` = 捲動過程中禁止使用者干預 |

#### 9.11.3 進階：modal / drawer 鎖背景捲動

開啟全螢幕對話框 / 側邊抽屜時，要鎖住背景捲動：

```vue
<script setup>
const { $lenis } = useNuxtApp()
const open = ref(false)

watch(open, (v) => {
  if (v) $lenis.stop()    // 鎖：滾輪/觸控/scrollTo 全部失效
  else   $lenis.start()   // 恢復
})
</script>
```

> [!CAUTION]
> 任何「鎖背景捲動」的 UI（modal、drawer、彈跳廣告、行動版選單）都**必須**呼叫 `$lenis.stop()`，**不要**用 `document.body.style.overflow = 'hidden'`。Lenis 是用 transform 模擬捲動，CSS `overflow: hidden` 無效，使用者仍會看到背景滑動。

#### 9.11.4 與 sticky / fixed 元素相容性

`position: sticky` 與 `position: fixed` 跟 Lenis **完全相容**，無需任何特殊處理。例如 [`Header01.vue`](../components/headers/Header01.vue) 的 `.header01 { position: sticky; top: 0 }` 在 Lenis 接管後仍能正常吸頂。

#### 9.11.5 第三方 modal / lightbox 鎖捲動

某些第三方套件（[LightGallery](https://www.lightgalleryjs.com/)、[Magnific Popup](https://dimsemenov.com/plugins/magnific-popup/) 等）開啟時會在 `<body>` 加 class 鎖背景。[`plugins/lenis.client.js`](../plugins/lenis.client.js) 用 `MutationObserver` 自動處理：

```js
const LOCK_BODY_CLASSES = ['overflow', 'popwinOverflow', 'loadingOverflow', 'lg-on']

const observer = new MutationObserver(() => {
  const locked = LOCK_BODY_CLASSES.some((cls) => document.body.classList.contains(cls))
  if (locked) lenis.stop()
  else lenis.start()
})
observer.observe(document.body, { attributeFilter: ['class'] })
```

如果你引入新的第三方套件且它用其他 class 名稱鎖捲動，請把該 class **加進 `LOCK_BODY_CLASSES` 陣列**。

#### 9.11.6 GSAP / ScrollTrigger 用法

GSAP 與 ScrollTrigger 已預先註冊，元件內可直接從 `useNuxtApp()` 取出：

```vue
<script setup>
const { $gsap, $ScrollTrigger } = useNuxtApp()
const target = ref(null)

onMounted(() => {
  $gsap.from(target.value, {
    opacity: 0,
    y: 60,
    duration: 1,
    scrollTrigger: {
      trigger: target.value,
      start: 'top 80%',   // 元素頂端碰到視窗 80% 位置時觸發
      end: 'bottom 20%',
      toggleActions: 'play none none reverse',
    },
  })
})

// 元件卸載時清掉這個元件相關的 ScrollTrigger，避免 memory leak
onBeforeUnmount(() => {
  $ScrollTrigger.getAll().forEach((st) => {
    if (st.vars.trigger === target.value) st.kill()
  })
})
</script>

<template>
  <section ref="target">滑進視窗時淡入</section>
</template>
```

> [!CAUTION]
> **ScrollTrigger 一定要在 `onMounted` 內呼叫**（不能在 `<script setup>` 頂層），因為它需要 DOM 已渲染。元件卸載時也要 `kill()` 對應的 ScrollTrigger 實例，否則路由切換後舊 trigger 仍在計算位置，會造成效能問題或錯誤。

### 9.12 全站樣式 hook：`data-project` / `data-page`

本專案有兩個自動掛在頁面上的 HTML 屬性，用來讓 SCSS 針對「整站類型」或「個別頁面」切換樣式 —— **元件不需手動加 class，這兩個屬性永遠是最新的**。

| 屬性 | 標在 | 值 | 由誰設定 |

| `data-page` | `<body>` | `index` / `news` / `about` / `contact` / … | [`app.vue`](../app.vue) 依當前路由第一段 |

**`data-page` 規則**（取路徑第一段；首頁為 `index`）：

| 路徑 | `data-page` |
|---|---|
| `/` | `index` |
| `/news` | `news` |
| `/news/master-launch` | `news`（同區共用） |
| `/about/company-intro` | `about` |
| `/shop/cart` | `shop` |
| `/firm/caption?unit=privacy` | `firm` |

**使用範例**（任何 `.vue` 的 `<style lang="scss" scoped>`）：

```scss
// 整站類型差異 — 例：購物站時 logo 字體變粗
[data-project="shop"] .logo { font-weight: 900; }

// 個別頁面差異 — 例：首頁 header 透明

  background: transparent;
}

// 整區共用樣式 — 例：news 區所有頁面的 banner 加暗色濾鏡
body[data-page="news"] .page-banner__bg { filter: brightness(0.7); }
```

> [!IMPORTANT]
> 寫元件樣式時**避免**自己手動往 `<html>` / `<body>` 加 class 或屬性 —— 統一從 `app.vue` 的 `useHead({ htmlAttrs, bodyAttrs })` 設定，集中管理避免衝突。需要新增頁面層級屬性時，去 `app.vue` 擴充。

### 9.13 `.env` boolean 值正確寫法

**`.env` 內所有值都是字串**，這是 JS 處理 boolean 環境變數最常見的坑 —— `"false"`、`" false"`（含空格）在 JS 裡都是 truthy（非空字串）！

```bash
# ❌ 錯誤 — 值前後有空格，會被當 truthy 字串
NUXT_PUBLIC_ENABLE_SHOP= false

# ❌ 錯誤 — 值是字串 "false"，!!"false" === true
# （但這個專案的 parseBool 會處理掉，所以這條算「險惡但救得回來」）

# ✅ 正確
NUXT_PUBLIC_ENABLE_SHOP=false
NUXT_PUBLIC_ENABLE_SHOP=true
```

#### 9.13.1 在 composable 內安全解析

切勿用 `!!pub.someFlag` —— 因為 `"false"`、`" false"` 都會被解析為 `true`。改用 [`composables/useProject.js`](../composables/useProject.js) 內定義的 `parseBool` 模式：

```js
const parseBool = (v) => typeof v === 'string'
  ? v.trim().toLowerCase() === 'true'
  : !!v

// 使用
enableShop: !!matrix.shop || parseBool(pub.enableShop),
```

**解析對照表**：

| `.env` 寫法 | `parseBool` 結果 |
|---|---|
| `NUXT_PUBLIC_X=true` | `true` |
| `NUXT_PUBLIC_X= true`（前空格） | `true` |
| `NUXT_PUBLIC_X=TRUE`（大寫） | `true` |
| `NUXT_PUBLIC_X=false` | `false` |
| `NUXT_PUBLIC_X= false`（前空格） | `false` |
| `NUXT_PUBLIC_X=`（空值） | `false` |
| 未設定 | `false` |

> [!CAUTION]
> 之後任何要當 boolean 用的 `NUXT_PUBLIC_*` 環境變數（debug 開關、功能旗標等）都**必須**走 `parseBool`，不要用 `!!`。

## 11. Header 版型切換

頁首支援多版型，透過設定切換，不需改動版面程式碼（footer 等多版型區塊亦可比照辦理）。

### 11.1 檔案結構

```text
components/
 ┣ AppHeader.vue        派發器（依設定選版型，不含設計）
 ┗ headers/
    ┣ Header01.vue          標準型（Logo 左、選單右）
    ┣ Header02.vue          置中型（Logo 置中、選單於下方）
    ┣ header_value.vue      VALUE 佳質食品風格
    ┗ <檔名>.vue            …（依需求擴充，檔名自取）
```

### 11.2 設定與切換

於 `.env` 指定版型 —— 值 = `headers/` 內的**檔名**（不含 `.vue`）：

```bash
# 值 = 版型檔名，例如 Header01、header_value
NUXT_PUBLIC_HEADER=header_value
```

切換版型只需改此值並重啟 dev server，版面程式碼完全不動。此設定與 `NUXT_PUBLIC_PROJECT_TYPE` 同屬 `runtimeConfig.public`，機制一致。

### 11.3 派發機制

`components/AppHeader.vue` 是薄派發器，以 Vite 的 `import.meta.glob` 自動載入 `headers/` 內全部版型，依 `NUXT_PUBLIC_HEADER` 選用對應檔；找不到時自動 fallback 第一個。

```javascript
const headers = import.meta.glob('./headers/*.vue', { eager: true })
const { public: pub } = useRuntimeConfig()
const current =
  headers[`./headers/${pub.header}.vue`]?.default ||
  Object.values(headers)[0]?.default
```

> ⚠️ 勿用 `resolveComponent('字串')` 取版型 —— Nuxt 自動引入元件無法以字串解析（同 `BlockRenderer` §8.2 的限制）。`import.meta.glob` 是正解。

### 11.4 新增版型

在 `headers/` 放一個 `.vue`（檔名自取）即可，再把 `NUXT_PUBLIC_HEADER` 設成該檔名；`AppHeader.vue` 不需修改。各版型**共用同一套資料 composable**（`useSiteMenu` / `useProject` / `useUiStore`），只有版面與樣式不同 —— 資料層不重複撰寫。

### 11.5 後台 Header 設定（`/admin/header`，全版型共用）

`/admin/header` 可視覺化調整以下項目，值存 `data/site-settings.json`，由 [`app.vue`](../app.vue) 依設定**注入 `<style>` 的 `!important` 規則**（不寫行內 `:style`），**全版型通用**（targeting `.app-header header` / `.app-header nav …`）：

- **Header 背景色**：版型預設 / 透明 / 指定色。
- **選單顏色（6 色）**：主選單文字 / 滑過、下拉文字 / 滑過、下拉背景 / 下拉單項背景。
  - 下拉用更深的選擇器（`nav [class*="__sub"] a`）→ **與主選單文字分開**：設了就覆寫、留空沿用主選單／版型預設。
  - 每色可「回到預設」，另有「全部回到預設」。
- LOGO 上傳 + 最大高度、自訂 CSS（注入 `<head>`）、語系顯示文字、navtool 6 icon 配置（per-Header）。

> 設定鍵見 `useEffectiveSettings`（state）、`site-settings.get/post`（白名單）、`useSiteSettings`（submit）。新增一個可調色＝這四處＋ `app.vue` 注入各加一筆。

## 12. Banner 版型切換

Banner 支援多版型（首頁輪播、內頁 hero），透過設定切換，不需改動頁面程式碼。機制與 [§11 Header 版型切換](#11-header-版型切換) 一致。

### 12.1 檔案結構

```text
components/
 ┣ PageBanner.vue            內頁 Banner 派發器
 ┣ BlockBanner.vue           首頁 Banner block 派發器
 ┗ banners/
    ┣ PageBanner01.vue        內頁 — 純背景圖 + 標題（現況）
    ┣ PageBannerXX.vue        內頁 — 其他版型（左圖右文、影片背景…）
    ┣ BlockBanner01.vue       首頁 — Swiper 複合型（主圖 fade + cover 同步 + YouTube + 新聞）
    ┗ BlockBannerXX.vue       首頁 — 其他輪播版型（Slide、Parallax…）
```

兩種派發器各自獨立，**不要混淆**：

| 派發器 | 用途 | 對應頁面 |
|---|---|---|
| `<PageBanner>` | 內頁 hero 區塊 | news / about / contact / faq / location / search / firm 等 |
| `<BlockBanner>` | 首頁輪播 block | 由 [`BlockRenderer.js`](../components/BlockRenderer.js) 在首頁派發 |

### 12.2 設定與切換

於 `.env` 指定版型 —— 值 = `banners/` 內的**檔名**（不含 `.vue`）：

```bash
# 內頁 Banner 版型
NUXT_PUBLIC_PAGE_BANNER=PageBanner01

# 首頁 Banner block 版型
NUXT_PUBLIC_BLOCK_BANNER=BlockBanner01
```

切換版型只需改此值並重啟 dev server，頁面程式碼完全不動。

### 12.3 派發機制

[`components/PageBanner.vue`](../components/PageBanner.vue) 與 [`components/BlockBanner.vue`](../components/BlockBanner.vue) 都是薄派發器，以 Vite 的 `import.meta.glob` 自動載入 `banners/` 內對應前綴的所有版型，找不到時自動 fallback 第一個：

```javascript
// PageBanner.vue
defineOptions({ inheritAttrs: false })
const variants = import.meta.glob('./banners/PageBanner*.vue', { eager: true })
const { public: pub } = useRuntimeConfig()
const current =
  variants[`./banners/${pub.pageBanner}.vue`]?.default ||
  Object.values(variants)[0]?.default
```

```vue
<template>
  <component :is="current" v-bind="$attrs" />
</template>
```

> [!IMPORTANT]
> **`defineOptions({ inheritAttrs: false })` + `v-bind="$attrs"`** 是關鍵 —— 讓所有 props（如 `page`、`title`、`subtitle`、`rows`）自動透傳給子版型。新增版型時派發器無需修改，只要子版型 `defineProps` 宣告對應參數即可接到值。

> 💡 **`BlockBanner` 例外：派發器自己取資料**。首頁 banner 內容走獨立 Banner API，`BlockBanner.vue` 在派發器內 `await useApiData('/banner/home')` 取整包（`rows` + 背景影片 `videoUrl`/`videoFile` + 跑馬燈 `news`）再傳給選定版型，不依賴 `/home/view` 聚合內嵌（見 [§8.1](#81-資料結構)）。

### 12.4 新增版型

在 `banners/` 放一個 `<PageBanner|BlockBanner>XX.vue`（XX 自取），檔名前綴須為 `PageBanner` 或 `BlockBanner`，再把對應的 `.env` 變數設成該檔名即可。各版型**共用同一份 API 來源**（`/banner/page`、`/banner/home`），只有版面與動畫不同 —— 資料層不重複撰寫。

範例（新增 `PageBanner02.vue` 左圖右文版）：

```vue
<script setup>
const props = defineProps({
  page: { type: String, default: '' },
  title: { type: String, default: '' },
  subtitle: { type: String, default: '' },
})

const { data: banners } = await useApiData('/banner/page', {
  key: `page-banner:${props.page}`,
  query: { page: props.page },
  default: () => [],
})

const bg = computed(() => banners.value?.[0]?.image?.pc || '')
</script>

<template>
  <section class="page-banner-02">
    <div class="page-banner-02__text">
      <h1>{{ title }}</h1>
      <p v-if="subtitle">{{ subtitle }}</p>
    </div>
    <img :src="bg" :alt="title" class="page-banner-02__img" />
  </section>
</template>
```

### 12.5 後台編輯（`/admin/banner`，dev mode）

首頁輪播（BlockBanner01）內容可在 `/admin/banner` 視覺化編輯，送出寫回 `banners.json`（即 `/banner/home`）：

- **多則 slide**：新增 / 刪除 / 排序；標題（第一則 `<h1>`、其餘 `<h2>`）/ 副標 / 說明文，可換行、可 **AI 生成**文案。
- **背景圖上傳**：選圖跳出**互動裁切器**（拖曳 / 縮放、比例固定），輸出固定 **2560 × 按比例高**，存到 `public/img/tw/banner/`。
- **圖片 `alt`（SEO）**：可手填，或 **AI 辨識**（Gemini Vision 看圖產生繁中替代文字）；前台空值會 fallback 用標題。
- **背景影片**：填 **YouTube 連結**；YT 留空時改**上傳影片檔**（HTML5、靜音循環、`playsinline`，**含 iOS 自動播放**），存 `public/video/banner/`。
- AI 功能需 `.env.local` 的 `NUXT_GEMINI_KEY`；端點 `/_admin/*` 由 `assertDev` 守門，`NODE_ENV=production` 直接 404。

`.env` 改 `NUXT_PUBLIC_PAGE_BANNER=PageBanner02` 並重啟 dev server，全站內頁 banner 立即套新版型。
