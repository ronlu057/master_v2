1. 概述

本文件描述形象 / 一般網站這部分的前台 API — 首頁、新聞、選單、聯絡表單、SEO、Banner、關於我們、多語等共 10 個 controller、約 5,000 行 PHP。

讀者：前端開發者（Vue / Nuxt / React / 行動 app）、UX 設計、QA、上手新團員。

1.1 與其他模組的職責分界

模組 範疇 文件
本冊 首頁聚合、新聞、選單、表單、SEO、Banner、About、I18n 本文件
購物系統 商品、購物車、結帳、付款、訂單、優惠券 購物系統白皮書
綠界整合 金流 / 物流 / 發票 SDK、後台操作、庫存圈存 綠界整合白皮書
WDM 後台 內容管理、商品管理、訂單管理 wdm/api/docs/*.md

1.2 設計原則

原則 說明
聚合勝過分散 首頁、選單等「一個畫面有很多 slot」的場景，後端一次回完整資料；前端 只負責 render，不負責重組
payload 結構穩定 每個 endpoint 的回應 schema 不變；前端可以放心預期欄位存在或為空陣列
fallback 一致 找不到時回空（[]、{}、null），不 throw — 讓前端可以 always render
語系自動切換 所有 query 自動帶 language = current_language，前端不用每次傳
軟刪除自動過濾 所有查詢自動帶 data_open = 1 AND data_trash = 0，下架的東西不會 leak

2. 共用約定

2.1 統一回應格式

成功：

{
  "status": "ok",
  "data": {
    "payload": <controller-specific data>,
    "meta":    { "lang": "tw", "time": "0.012s", ...action-specific... }
  }
}

失敗（throw 對應 exception 由 dispatcher 統一處理）：

{
  "status":  "error",
  "code":    "VALIDATION",
  "message": "...",
  "details": { ... }
}

2.2 語系

language 由 dispatcher 從 URL path 解析（/tw/api/... / /en/api/...）；無前綴則用 _WebUseLanguage[0]['lang_type']。Controller 透過 $this->language 取得。

2.3 軟刪除

所有 list / detail 查詢自動排除 data_trash = 1，不需前端傳參數。

2.4 圖片 URL 規則

回應內所有圖片欄位都是完整 URL（含 host），不是相對 path。前端拿到直接放進 <img src> 即可。

3. HomeApiController — 首頁 Layout 聚合

程式位置：api/Controllers/HomeApiController.php 規模：1,193 行 — 整個模組最大、最核心的 API

3.1 設計理念

首頁通常一頁裡有多個區塊（橫幅、CTA、推薦商品、新聞、合作夥伴…），如果每塊各打一支 API，會：

  • 多次 round-trip
  • 載入順序難掌握
  • 前端要組裝邏輯

本 controller 把首頁整頁需要的所有資料一次撈完、一次回。前端只要：

const home = await api('/home/view');
// 依照 payload 的 slots 順序 render 即可，不用做任何重組

3.2 主要端點

Endpoint Method 用途
/api/home/view GET 首頁整頁聚合（最重要）
/api/home/cipage GET 只回首頁框架 boxes（cipage 編輯器內容）
/api/home/homePush GET 推薦單元（push 區塊）
/api/home/homeOtherAd GET 其他廣告區（祝賀、行事曆等）
/api/home/customBlocks GET 自訂區塊（首頁標籤的新聞）

3.3 /home/view payload 結構

最外層是 slots 陣列，每個 slot 來自不同資料來源：

{
  "slots": [
    {
      "key":    "cipage",
      "type":   "slot",
      "blocks": [...]
    },
    {
      "key":    "homePush",
      "type":   "slot",
      "blocks": [...]
    },
    {
      "key":    "homeOtherAd",
      "type":   "slot",
      "blocks": [...]
    },
    {
      "key":    "news_block",
      "type":   "slot",
      "blocks": [...]
    }
  ]
}

3.4 block 通用格式

每個 slot 內的 block 都長這樣：

{
  "type":  "<block-type>",
  "title": "<可選的區塊標題>",
  "item": {
    "rows": [
      { ... 一筆資料 ... },
      { ... 一筆資料 ... }
    ]
  }
}

關鍵約定：

  • 所有 block 的資料一律放在 block.item.rows（不會出現 block.data / block.list 等不一致 key）
  • 沒資料的 block 不會出現在 blocks 陣列（不是回空 rows）
  • 前端只需用 block.type 決定要套哪個 template，內容固定從 block.item.rows 取

3.5 news_block 特殊欄位

新聞區塊的每筆 row 多帶幾個常用欄位：

{
  "url":              "/news/article-blink",
  "title":            "文章標題",
  "date":             "5.21,2026",          // 預格式化好
  "weekday":          "Wed",
  "category_blinks":  ["news", "events"],   // 分類麵包屑
  "allPage":          "/news?category=news",
  "rows":             [ ...更多 sub-rows... ]
}

3.6 主要資料來源表

cipage / system_home_other_ad / system_items_file / system_relative_list / news_category / news

3.7 前端 render pseudo-code

const home = await api('/home/view');
home.slots.forEach(slot => {
  slot.blocks.forEach(block => {
    const Component = BLOCK_COMPONENTS[block.type];
    if (!Component) return; // 未知 type 直接 skip
    render(Component, { title: block.title, rows: block.item.rows });
  });
});

4. SearchApiController — 全站跨模組搜尋

程式位置：api/Controllers/SearchApiController.php 規模：921 行

4.1 設計理念

「搜尋框打一個字 → 拿到全站所有相關結果（新聞、商品、頁面…）」這種跨模組搜尋，由本 controller 統一處理。

各模組的 searchability 由後台 db_system_unit_info 設定，本 controller 動態讀取後逐表搜尋。

4.2 端點

GET /api/search?keyword=xxx&page=1&limit=20[&url=xxx]

INPUT_RULES：keyword, page, limit, url

4.3 回應結構

{
  "keyword":    "搜尋字",
  "categories": [
    { "unit_id": 1, "type": "news",  "title": "新聞",   "count": 23 },
    { "unit_id": 5, "type": "shop",  "title": "商品",   "count": 8 },
    { "unit_id": 9, "type": "about", "title": "關於我們", "count": 2 }
  ],
  "list": [
    {
      "unit_id":         1,
      "type":            "news",
      "type_title":      "新聞",
      "title":           "結果標題",
      "url":             "/news/xxx",
      "category_blinks": ["news"],
      "allPage":         "/news",
      "has_detail":      true,
      "page_button":     "查看",
      "target":          "_self",
      "summary":         "節錄文字..."
    }
  ]
}

4.4 前端 UI 建議

  • 上方 categories 變成 tab / chip 篩選器，秀每類筆數
  • 下方 list 用 type 切換顯示樣式（新聞顯示日期、商品顯示價格...）
  • has_detail = false 的項目可能只是分類連結（用 page_button 文字）

4.5 輕量 / 進階端點

Endpoint Method 用途
/api/search/suggest GET 搜尋框自動完成 / 推薦。輕量版：只回資料項（標題 + 連結），不算 categories / summary / 分頁，求快（limit 上限 20，預設 10）
/api/search/semantic GET 語意搜尋（Stage C）。把使用者口語需求丟 AI 即時抽關鍵詞 → 同義詞展開 → 走跟 suggest 同一套搜尋排序

/api/search/suggest：參數同 /api/search（keyword / limit / url），回 { keyword, terms, suggestions[] }。

/api/search/semantic：參數 q（自然語言，亦接 keyword）/ limit / url。

  • 公開免 token（同 suggest），但因為會打 AI（有成本）所以：① 每 IP 限流（預設 20 req / 60s，超過回 429）；② 抽字結果做檔案快取（同一句話一天內不重打 AI）
  • AI 走 AI_PROVIDER_PRIORITY（gemini → ollama → anthropic）失敗自動遞補；全掛 → 退化成把原句直接當關鍵字搜（不整支死掉）
  • 回 { query, extracted[], terms[], suggestions[], provider, cached }
  • 詳細 AI 串接見 AI 串接文件

6. MenuApiController — 選單

程式位置：api/Controllers/MenuApiController.php 規模：769 行

6.1 設計理念

支援多位置選單（header / footer / mobile / quick）+ 多階層 children + 下拉選單（可帶資料）+ 自訂連結。

WDM 後台用 db_system_unit_info 管理每個選單項，本 controller 把它轉成前端易用的結構。

6.2 端點

Endpoint Method 用途
/api/menu/view GET 前端 render 用（推薦） — 已 flatten 成標準 children 樹
/api/menu/header GET 只回 header 選單（含 dropdown）
/api/menu/menu GET 所有位置（raw 結構，含 dropdown layout）

6.3 /menu/view 回應（前端用）

{
  "header": [
    {
      "title":     "關於我們",
      "data_type": "about",
      "url":       "/about",
      "children": [
        { "title": "公司簡介", "url": "/about/intro", "children": [] },
        { "title": "團隊", "url": "/about/team", "children": [] }
      ]
    }
  ],
  "footer": [ ... ],
  "mobile": [ ... ]
}

6.4 Dropdown 模式

後台可為每個選單項設定 dropdown 型態：

  • none — 無下拉
  • simple — 純連結列表
  • with_data — 把該分類底下最多 5 筆資料當 children 接上（例：在「商品」選單懸停顯示前 5 個熱門商品）

/menu/view 已經把 with_data 結果攤平到 children，前端不用區分。

6.5 自訂連結

每個位置都可在後台加「自訂連結區塊」（如：footer 加「客服 LINE」「FB 粉專」），會出現在 custom_links 陣列（/menu/menu raw 才看得到；/menu/view 已併入 children）。

7. ContactApiController — 表單 / 訂閱 / 退訂

程式位置：api/Controllers/ContactApiController.php 規模：505 行

7.1 端點

Endpoint Method 用途
/api/contact/csrf GET 取 CSRF token（表單送出前必拿）
/api/contact/submit POST 一般聯絡表單
/api/contact/subscribe POST 訂閱電子報
/api/contact/unsubscribe POST 申請退訂（寄 token 信）
/api/contact/unsubscribe GET 點信中連結確認退訂（帶 ?token=）

7.2 欄位規格

submit（一般聯絡表單）

欄位 必填 規則
name ✅ 姓名
tel ✅ 6–20 位數字
mail ✅ email
theme ✅ 主題
comment ✅ 內容
gender ❌ 0 / 1
company ❌ 公司名
address ❌ 地址
csrf_name + csrf_value ✅ 由 /csrf 取得

subscribe（訂閱）

firstname / lastname / title / affiliation / mail 全部必填。

unsubscribe_request（退訂申請）

mail 必填、suggest 選填。

7.3 退訂的雙階段流程

顧客在退訂頁送出 mail
        ↓
POST /api/contact/unsubscribe { mail, suggest }
        ↓
系統產生退訂 token + 寄確認信
        ↓
顧客收信點裡面的連結
        ↓
GET /api/contact/unsubscribe?token=xxx
        ↓
真正執行退訂

設計理由：避免有心人士拿別人 email 亂退。

7.4 回應

成功：{ "success": true }

失敗會帶 reason（如 mail_send_failed）方便前端提示。

7.5 主要資料表

db_contact / db_contact_info / db_unsubscribe_token

8. SeoApiController — SEO Meta

程式位置：api/Controllers/SeoApiController.php 規模：207 行

8.1 端點

Endpoint Method 用途
/api/seo/list?page_type=xxx GET 列表 / 詳細頁通用 SEO meta
/api/seo/home GET 首頁專屬 meta（含 OG、author / copyright / robots）

8.2 fallback 機制

當 page_type 沒設 SEO 時，自動 fallback 到 common 預設。回應會多帶兩個欄位讓前端知道來源：

{
  "title":           "...",
  "description":     "...",
  "keywords":        "...",
  "head_script":     "...",   // GTM / GA 等 head 追蹤碼
  "body_script":     "...",   // body 追蹤碼
  "source":          "page|common",
  "fallback_reason": "page_not_found|empty|null"
}

8.3 home 額外欄位

{
  "og": {
    "title":       "...",
    "description": "...",
    "image":       "https://..."
  },
  "meta": {
    "author":       "...",
    "copyright":    "...",
    "distribution": "...",
    "rating":       "...",
    "robots":       "..."
  }
}

8.4 前端整合

Nuxt / Next：在 useHead / Head 把這支回來的欄位塞進去就好；head_script / body_script 用 v-html 或對應的 inject 機制注入。

8.5 結構化資料（JSON-LD）/ sitemap / robots.txt — 由前端負責

這三項後端不產生，由前端框架靜態 / SSR 生成：

項目 推薦做法（Nuxt 3 範例）
JSON-LD（Product / Article / Organization 等） 前端拿 detail API 後，組 schema.org 物件 → useHead({ script: [{ type: 'application/ld+json', children: JSON.stringify(...) }] })
sitemap.xml 用 @nuxtjs/sitemap 模組；拉 /api/shop/list + /api/news/list + /api/article/list 等 build-time / runtime 動態產生
robots.txt 用 @nuxtjs/robots 模組；正式 / 測試環境給不同 policy（dev → Disallow: /、prod → Allow: /）

後端目前不提供 /sitemap.xml 或 /robots.txt endpoint —— 因為這兩個是「全站 metadata」由前端框架更知道路由結構。若要加，可以在 .htaccess 路由到對應的前端 SSR endpoint。

Firm 資料的結構化標記：/api/firm/view 回應裡有 schema.localbusiness 欄位（從 db_firm.data_schema_localbusiness 來），是後台手動編輯的 JSON-LD 字串。前端拿到後直接塞到 <script type="application/ld+json"> 即可 — 不要 stringify 再 stringify。

9.4 I18nApiController — 多語字典

GET /api/i18n

回前端用的 key → value 字典（JSON 物件），翻譯空白時自動 fallback 為 key 本身（避免畫面破版）：

{
  "nav.home":      "首頁",
  "nav.about":     "關於我們",
  "btn.submit":    "送出",
  "msg.success":   "操作成功"
}

不依賴 DB，讀自 _WebTranslateLangPath 設定的檔案。

另有取語系清單的端點：

GET /api/i18n/languages

回本站開啟的全部語系清單，每筆含 lang_type / lang_name / html_tag_lang / currency / currency_symbol / is_default / hide_from_menu，給語系切換選單用。

13. 前端整合範例（v1.7 新增）

幾個常用 pattern 的 JS 範例 — FE 拿來改就能用，不必每個 module 都重發明輪子。

13.1 通用 list 分頁

所有 list 類 endpoint（News / FAQ / Download / Location / Product / Article ...）都遵守同一個分頁約定，FE 可以寫一個通用函式：

async function fetchList(endpoint, { page = 1, limit = 20, ...filters } = {}) {
  const qs = new URLSearchParams({ page, limit, ...filters });
  const r = await fetch(`${endpoint}?${qs}`).then(r => r.json());
  return {
    items: r.data.payload,
    page:  r.data.meta.page,
    total: r.data.meta.total,
    totalPages: r.data.meta.total_pages,
  };
}

// 例：拉新聞第 2 頁，類別=活動
const news = await fetchList('/api/news/list', { page: 2, category: 'event' });

13.2 通用 detail（含 SEO meta 跨模組共通）

凡是有 detail endpoint 的模組（News/Faq/Download/Article/Esg/Product 等）都會回類似結構，FE 可以用同一個元件 render：

async function fetchDetail(endpoint, blink) {
  const r = await fetch(`${endpoint}?blink=${encodeURIComponent(blink)}`).then(r => r.json());
  if (r.status === 'error' && r.code === 'NOT_FOUND') throw new Error('not_found');
  const p = r.data.payload;
  return {
    title:   p.title,
    content: p.content || p.memo2 || p.detail_html,
    images:  p.images || [p.image].filter(Boolean),
    meta:    p.meta,   // 給 <Helmet> 或 next/head 寫 SEO
  };
}

13.3 Contact 表單（CSRF 兩步）

聯絡 / 訂閱 / 退訂都需要先拿 CSRF token 才能 submit：

async function submitContactForm(form) {
  // Step 1: 拿 CSRF token
  const csrf = await fetch('/api/contact/csrf').then(r => r.json().then(j => j.data.payload.csrf_token));

  // Step 2: 帶 token 一起 submit
  const r = await fetch('/api/contact/submit', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ ...form, csrf_token: csrf }),
  }).then(r => r.json());

  if (r.status !== 'ok') throw new Error(r.message);
  return r.data.payload;
}

13.4 彈跳廣告（進站只跳一次）

async function showEntryPopups() {
  const popups = await fetch('/api/popupsAd/list').then(r => r.json().then(j => j.data.payload));
  const dismissed = JSON.parse(localStorage.getItem('dismissed_popups') || '[]');

  for (const p of popups) {
    if (dismissed.includes(p.id)) continue;   // 使用者已關掉
    showModal(p);
    dismissed.push(p.id);
  }
  localStorage.setItem('dismissed_popups', JSON.stringify(dismissed));
}

13.5 報名（業務狀態檢查在前端先過濾）

/api/enroll/list 後端會在每筆活動加 is_open / is_expired / is_full / can_enroll 旗標：

const acts = await fetchList('/api/enroll/list', { category });
acts.items.forEach(a => {
  if (!a.can_enroll) {
    // disable 報名按鈕；顯示原因（已滿 / 過期 / 未開放）
    const reason = a.is_full ? '已額滿' : a.is_expired ? '已截止' : '未開放';
    renderClosedActivity(a, reason);
    return;
  }
  renderOpenActivity(a);
});

13.6 報修狀態追蹤（雙因子）

async function trackRepair(repairNo, email) {
  const qs = new URLSearchParams({ number: repairNo, mail: email });
  const r = await fetch(`/api/repair/status?${qs}`).then(r => r.json());
  if (r.status === 'error' && r.code === 'REPAIR_NOT_FOUND') {
    // 後端不洩漏「編號存不存在」/「email 對不對」的差別 — UI 一律顯示「查無」
    return null;
  }
  return r.data.payload;   // { audit, audit_label, ... }
}

13.7 統一錯誤處理

形象站 endpoint 的錯誤回應都是統一 envelope：

{
  "status": "error",
  "code": "INVALID_PARAM",
  "message": "缺少必填：name",
  "meta": { "request_id": "...", "time": "..." }
}

FE 用統一 wrapper：

async function apiCall(path, opts = {}) {
  const r = await fetch(path, opts);
  const j = await r.json();
  if (j.status === 'ok') return j.data.payload;
  // 後端 code 已分類（INVALID_PARAM / NOT_FOUND / FORBIDDEN / SERVER_ERROR ...）
  const err = new Error(j.message);
  err.code = j.code;
  err.requestId = j.meta?.request_id;
  throw err;
}
