1.4 五分鐘理解整個專案結構

master_v2/
├─ pages/              📄 檔案路由（檔名 = 網址）
├─ components/         🧩 共用元件
│  ├─ headers/         Header 版型集合
│  ├─ banners/         Banner 版型集合
│  ├─ AppHeader.vue    Header 派發器
│  ├─ PageBanner.vue   PageBanner 派發器
│  └─ ...
├─ composables/        🪝 共用邏輯（自動引入，不用 import）
├─ stores/             🍍 Pinia（全站狀態）
├─ assets/
│  ├─ styles/          主題色 / SCSS 變數
│  └─ icon/            SVG icon
├─ public/             原樣輸出的資源
├─ server/mock/data/   假後端資料（前端開發階段用）
├─ plugins/            Lenis / GSAP 等全站初始化
├─ app.vue             進入點
├─ app.config.js       五種專案類型功能矩陣
├─ nuxt.config.js      Nuxt 全局設定（字體、SCSS、modules）
└─ .env                專案類型 / Header / Banner / API 設定

關鍵概念：「一套程式碼 + 設定 = 不同的網站」

  • 改 .env 的 NUXT_PUBLIC_PROJECT_TYPE → 整站行為變
  • 改 .env 的 NUXT_PUBLIC_HEADER → 換頁首版型
  • 改 .env 的 NUXT_PUBLIC_PAGE_BANNER → 換內頁 Banner

1.5 練習一：改 mock 資料

目標：改首頁標題、改聯絡資訊。

  1. 開 server/mock/data/firm.json
  2. 把 firm.title 從「AD-Design 母版示範站」改成你想要的
  3. 把 firm.tel 改成 0912-345-678
  4. 存檔，重啟 dev server（Ctrl+C → npm run dev）
  5. 重新整理瀏覽器，看標題與聯絡資訊變了

改 JSON 必須重啟 dev server，因為 mock 不在 HMR 範圍。

1.6 練習二：切換 Header 版型

目標：把 Header01 切換成 Header02 看效果差異。

  1. 開 .env
  2. 改 NUXT_PUBLIC_HEADER=Header02
  3. 重啟 dev server
  4. 觀察 logo 位置、選單樣式、社群圖示的差異

接著你可以試試自己建 Header03.vue（複製 Header01.vue 改個 class 與顏色），改 .env 指向 Header03，馬上看到自己的作品。

1.7 練習三：新增一個頁面

目標：新增「最新活動」頁面 /events。

  1. 在 pages/ 建 events.vue：
<script setup>
useHead({ title: '最新活動' })
</script>

<template>
  <div>
    <PageBanner page="events" title="最新活動" subtitle="一年一度盛大開跑" />
    <div class="container">
      <h2 class="section-title__main">本季精選</h2>
      <p>內容 placeholder...</p>
    </div>
  </div>
</template>
  1. 在 server/mock/data/menu.json 的 header 陣列加：
{ "title": "最新活動", "url": "/events", "data_type": "page", "children": [] }
  1. 重啟 dev server，點擊主選單的「最新活動」可以進到你的頁面。

Part 3 — 設計師 / PM 指南

不寫程式也能看懂：這套網站，哪裡可以調？怎麼跟工程師討論需求？

3.1 一句話講懂本專案

「一份程式碼，五種專案類型，靠設定值切換」 —— 工程師寫的程式可以同時做出五種不同的網站。

3.2 五種專案類型

類型 設定值 適合什麼網站
模組件套版 module 標準形象站、最快上線
客製首頁 custom-home 首頁要特別、內頁標準
全部客製 full-custom 整站獨特設計
購物類型 shop 電商站，含購物車
臨時站 temp 「即將開幕」、活動過渡頁

選擇邏輯：

  • 預算 / 時程緊 → module（最便宜、最快）
  • 首頁要做出特色 → custom-home
  • 整站不能將就 → full-custom
  • 要賣東西 → shop
  • 短期使用 → temp

3.3 哪些「可以調」（不需要重寫）

3.3.1 視覺主題

  • ✅ 主色、輔色、文字色、背景色（CSS 變數一改全站變）
  • ✅ 圓角、陰影、間距、容器寬度（CSS 變數）
  • ✅ 字體（從 Google Fonts 選任何字型）

3.3.2 版型切換（一個 .env 設定即可換）

  • ✅ Header 版型（已有 3 套：標準型 / 置中型 / VALUE 風格）
  • ✅ 內頁 Banner 版型
  • ✅ 首頁輪播 Banner 版型

3.3.3 功能開關（一個 .env 設定即可開關）

  • ✅ 購物車（即使不是 shop 類型也能加掛）
  • ✅ 多語切換 — 目前內建 4 語系（繁中 / 英 / 日 / 韓），工程師可加減（自動判斷剩 1 個時隱藏 icon）
  • ✅ 預設語系（.env 的 NUXT_PUBLIC_DEFAULT_LANG，使用者第一次來看到的語言）
  • ✅ 搜尋功能

3.3.4 內容（後端 / 後台調整）

  • ✅ 所有文字、圖片、選單項
  • ✅ 公司資訊、社群連結
  • ✅ Banner 內容（圖片、文字、連結）
  • ✅ FAQ、據點、下載中心、新聞

3.4 哪些「不可以調」（會牽動結構）

❌ 改動成本高、要工程師評估時程

  • 把「客製首頁」中途改成「全部客製」（首頁要重新手刻）
  • 改網站的根本路由結構（如 /news 改成 /blog）
  • 改 mock 資料的「結構」（不是內容） —— 例如把選單從兩層變三層

❌ 工程師會勸退（建議走標準功能）

  • 「我要每頁有不一樣的 Header」 —— 多版型機制是好，但建議保持一致
  • 「我要客戶後台自己換 logo」 —— 那是後台功能，不在前端母版範圍
  • 「我要這顆按鈕做特殊動畫」 —— 可以做，但會吃進度，建議用既有的 hover transition

3.5 跟工程師討論需求時

高效溝通的關鍵字

✅ 用這套詞彙跟工程師描述：

想說 用這個詞
整個頁面的大圖 Banner（首頁輪播叫 BlockBanner，內頁叫 PageBanner）
頁面上方的選單列 Header
頁面下方的版權區 Footer
首頁上一塊一塊內容 Block 區塊（如 BlockNews、BlockIntro）
點選後跳出來的視窗 Modal
滑鼠移過去才出現的選單 Dropdown / Popup
一塊塊小區域的標題 + 圖 卡片 / Card

確認需求時必問

  1. 是哪個專案類型？（module / custom-home / full-custom / shop / temp）
  2. 要不要購物功能？（NUXT_PUBLIC_ENABLE_SHOP）
  3. 要不要多語？（中、英、日等）
  4. Header 用哪種？（給設計稿前先決定版型，避免做完才發現要換）

設計稿交付給工程師時

✅ DO：

  • 提供 desktop / tablet / mobile 三個尺寸
  • 用 Figma / XD 等工具標示間距、字級
  • 把所有 hover / active / focus 狀態都畫出來
  • 明確列出每個 block 的順序

❌ DON'T：

  • 只給一張 JPG 就要工程師「按這做」（吃資源、容易誤解）
  • 用「跟那個 OO 網站很像但不一樣」描述（請給具體規格）
  • 期待工程師逐像素 1:1 還原（白皮書 §9.8：精準視覺由人工調整，AI 處理結構與邏輯）

3.6 從報價到上線，流程是什麼？

1. 需求確認         決定專案類型、Header / Banner 版型、是否購物
2. 設計階段         設計師出設計稿、PM 整理需求文件
3. mock 階段        工程師建 mock JSON + 套版（看得到畫面但無真實資料）
4. 後端串接         接真實 API，從 .env 切換 API_BASE
5. 內容上稿         客戶 / 編輯把實際內容輸入後台
6. 測試 / 修整      QA、跨瀏覽器測試、效能調整
7. 部署上線         build 後部署到正式環境

3.7 看到工程師說這些，意思是

工程師說 翻譯
「這要重啟 dev server」 改了設定檔（.env / nuxt.config.js），要重新啟動才生效
「這個是 mock 資料」 還沒接真實後端，先用假資料測畫面
「白皮書沒寫」 沒有預先設計這個機制，要評估改動範圍
「會牽動 SSR」 改的是伺服器端渲染邏輯，要小心，可能影響 SEO
「需要 rebuild」 改了某些設定，要重新編譯整個專案
